# 架构重构复盘报告（个人知识服务）

## 1. 现有执行偏差复盘

### 1.1 功能模块未与大模型联动的表现（重构前）
- 规则管理、文档上传、实验运行均以独立按钮触发，用户需手工判断调用顺序。
- 推理链路主要由固定流程执行，缺少「模型驱动的需求拆解与工具编排」。
- 结果输出更偏结构化日志，缺少模型层语义总结与行动建议。

### 1.2 未接入向量检索的痛点（重构前）
- 只能基于结构字段与全量状态查看，不支持语义召回。
- 复杂文档问答时需要人工从知识列表中定位相关片段。
- 随知识规模增长，检索延迟和精准度下降，难以支持万级条目。

### 1.3 知识与模型能力割裂场景（重构前）
- 文档能入库，但用户问题无法自动形成“检索→推理→行动”闭环。
- 决策约束输出后仍需用户二次解释给业务场景。
- 营销客服场景中，制度知识与回复策略未形成统一智能流程。

---

## 2. 重构后目标架构（模型为脑、知识为基、功能为手脚）

- **模型脑（Brain）**：`PersonalKnowledgeAssistant`
  - 负责需求理解、工具调度、结果汇总与对话输出。
- **知识基（Knowledge Base）**：`LocalVectorDB + KnowledgeWeightedRetriever`
  - DOC/PDF 解析后文本进入向量索引；检索结果按“语义相似+置信度+来源权重-冲突惩罚”重排。
- **功能手脚（Toolkit）**：文档上传、向量检索、认知循环、遥测读取
  - 所有能力通过 `BrainToolkit` 被模型间接调用。

该架构保持并强化了原有知识创新机制（冲突、置信、来源加权），避免被纯 LLM 黑箱覆盖。

---

## 3. 功能调用指令集（标准化）

| Tool | Trigger | 参数 | 返回 |
|---|---|---|---|
| `upload_document` | 用户上传 DOC/PDF | `filename, content_base64, scenario, source` | `status, metadata, ingested, skipped` |
| `retrieve_knowledge` | 用户提问或模型需要证据 | `query, top_k` | `id,score,topic,text` |
| `run_cognition` | 问题包含建议/策略/决策，或营销客服场景 | `goal, scenario` | `decisions, diagnostics` |
| `load_telemetry` | 用户请求复核/审计 | `none` | `documents, judgements, loop_runs` |

## 4. 三轮迭代记录（闭环）

### Round 1：偏差修正
- 问题：DOC/PDF 上传后难以直接形成语义问答。
- 动作：增加向量索引 + 助手统一入口 `/api/assistant/query`。
- 结果：个人用户从“多按钮串联”变为“上传文档+自然语言需求”双步闭环。

### Round 2：营销客服助手优化
- 问题：制度检索慢、回复口径不一致。
- 动作：增加营销案例基线对比接口；模型输出要求包含“结论+依据+行动”。
- 结果：可形成可审计的政策回复策略。

### Round 3：9场景泛化
- 问题：跨场景稳定性需要量化。
- 动作：9个角色大样本（>1000）统一实验，输出 pre/post 指标。
- 结果：形成可复现的对比报告并定位不可用场景继续优化。

---

## 5. 验证结论
- 当前已满足：
  - 文档上传（DOC/PDF）→ 向量检索 → 大模型推理输出闭环。
  - 功能能力可作为工具由模型调度（通过 Assistant + Toolkit）。
  - 9场景大样本验证与三轮报告输出。
- 下一步建议：
  - 对接生产级向量库（Milvus/Chroma/Pinecone）以提升万级召回性能。
  - 引入在线评测集，确保语义匹配准确率持续≥90%。
