# MemF 下一阶段落地方案（管理端 + 示例场景验证）

> 背景：当前核心功能已实现，但缺少可视化管理与可复现实验，导致“功能是否正确、业务价值是否成立”无法被快速验证。

## 1. 任务一：建设页面端管理控制与可视化能力

### 1.1 目标
- 给研发/运营/业务同学提供统一入口，完成“知识接入 → 索引构建 → 检索调试 → 日志追踪”闭环。
- 支持按用户（或租户）隔离知识，满足“用户绑定自己的知识”诉求。

### 1.2 建议页面结构
1. **仪表盘（Dashboard）**
   - 指标卡片：知识源数量、文档数量、切片数量、近24h请求量、平均响应时间、成功率
   - 趋势图：查询量趋势、错误率趋势、索引更新频率

2. **知识源管理（Sources）**
   - 新增知识源：文件上传、URL抓取、API导入
   - 操作：启停、重同步、删除、查看最近同步结果
   - 字段建议：source_id、source_name、owner_user_id、status、last_sync_at、doc_count

3. **知识库内容（Knowledge Base）**
   - 文档列表 + 切片预览
   - 支持按 `user_id`/标签/时间筛选
   - 可查看文档切片后的实际召回内容（便于排查召回不准）

4. **问答调试台（Playground）**
   - 输入问题后展示：
     - 检索 TopK 片段
     - 重排分数（如有）
     - 最终生成答案
     - 总耗时 + 各阶段耗时（召回、重排、生成）

5. **运行日志（Logs/Tracing）**
   - 每条请求记录：query、召回片段ID、回答、token消耗、耗时、错误信息
   - 支持导出 CSV，便于离线评估

### 1.3 最小接口清单（建议）
- `POST /api/sources`：创建知识源
- `GET /api/sources`：查询知识源
- `POST /api/sources/{id}/sync`：触发同步/切片/建索引
- `GET /api/knowledge/documents`：分页查询文档
- `POST /api/query/debug`：执行检索+生成，返回中间链路信息
- `GET /api/logs/queries`：查询请求日志

### 1.4 用户绑定知识（多租户）
- 数据层强制包含 `user_id`（或 `tenant_id`）维度。
- 查询链路默认使用当前登录用户的知识空间。
- 管理员角色可切换查看所有用户，普通用户仅可见自身数据。

### 1.5 验收标准
- 能通过页面完成一次完整流程：
  1) 创建用户/选择用户
  2) 绑定知识并同步
  3) 在调试台提问
  4) 在日志页看到完整链路
- 同一问题在不同用户下返回的知识引用不同（隔离生效）。

## 2. 任务二：确定示例场景并完成效果验证

## 2.1 推荐先做“金融知识助手”场景（更容易体现价值）

### 场景定义
- 目标用户：理财顾问/客服人员
- 知识范围：基金产品说明、费率规则、风险揭示、适当性条款
- 典型问题：
  - “某产品申购费率是多少？”
  - “风险等级R3客户能否购买该产品？”
  - “赎回到账时间如何计算？”

### 为什么适合做首个示例
- 文档结构化程度高，便于建立标准答案。
- 对“准确率、可追溯性、时效性”要求高，能体现系统优势。
- 业务价值可量化（减少人工检索时间、降低误答风险）。

## 2.2 示例数据与实验资产组织建议

建议新增目录：

```text
examples/
  finance-assistant/
    data/
      raw_docs/           # 原始文档（pdf/markdown）
      qa_set.jsonl        # 问题-参考答案-证据片段
    scripts/
      run_eval.py         # 批量评测脚本
    report.md             # 结果报告
```

## 2.3 评估指标（先小而全）
- **检索指标**：Recall@5、MRR
- **回答指标**：
  - 事实一致性（是否与证据冲突）
  - 关键点覆盖率（命中参考答案要点）
- **工程指标**：P95响应时间、失败率

## 2.4 对比实验设计
- Baseline：无知识检索（仅模型直答）
- MemF 当前实现：检索增强回答
- （可选）MemF + 重排/提示词优化：进一步提升准确率

至少输出以下结果：
1. 三组方案的准确率/覆盖率/耗时对比表
2. 10 条典型问答案例（含正确与失败样例）
3. 失败原因归因（召回偏差/知识过期/提示词不稳定）

## 2.5 业务价值描述模板（可直接用于汇报）
- 检索耗时从 X 分钟降低到 Y 秒。
- 高频问答准确率提升 Z%。
- 新人客服上手时间减少 N 天。

## 3. 分阶段执行建议（4 周）

### 第 1 周
- 确认数据模型（含 `user_id` 隔离）
- 打通知识源管理 + 同步接口
- 完成最小版 Dashboard

### 第 2 周
- 完成调试台与日志链路
- 接入金融示例数据集（首批 30~50 条问题）

### 第 3 周
- 跑通批量评测脚本，产出第一版 report
- 根据失败案例优化召回参数

### 第 4 周
- 完成演示脚本与业务汇报材料
- 固化“新增场景模板”（便于后续扩展到 AI 场景）

## 4. 你现在就可以执行的最小动作（本周）
1. 先定一个示例场景：`finance-assistant`。
2. 准备 20 份文档 + 50 个标准问题。
3. 实现调试台最小功能：显示“检索片段 + 最终答案 + 耗时”。
4. 每天看日志，先把 Top3 错误类型归因出来。

这样你就能快速判断：
- 功能是否“真的可用”；
- 项目是否“真的有业务价值”。
